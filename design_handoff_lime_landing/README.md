# Handoff: ЛАЙМ — лендинг (CRM-внедрение × вайбкодинг)

## Overview
Одностраничный маркетинговый лендинг студии «ЛАЙМ», которая занимается внедрением CRM (amoCRM, Битрикс24), интеграциями, ИИ-агентами, кастомной разработкой на «вайбкодинге», своими SaaS и MVP. Визуальный язык вдохновлён сайтом Hut 8 (Awwwards): светлая «бумажная» палитра, тонкие линии, много воздуха, премиальная типографика и **анимированный изометрический «живой город» на фоне, который пустой в начале и постепенно достраивается в единый механизм по мере скролла** — метафора «наше внедрение улучшает бизнес».

## About the Design Files
Файлы в этом бандле — **дизайн-референс, сделанный в HTML**. Это прототип, демонстрирующий задуманный внешний вид и поведение, **а не продакшн-код для прямого копирования**. Задача — **воссоздать этот дизайн в целевом окружении проекта** (React / Next.js / Vue / Svelte и т.д.) по принятым в нём паттернам и библиотекам. Если кодовой базы ещё нет — выбрать подходящий фреймворк (рекомендация: **Next.js + React**, т.к. интерактив завязан на canvas и scroll) и реализовать дизайн там.

Главный нюанс реализации: фоновый «город» — это **single `<canvas>` 2D-рендер с собственным изометрическим движком**, управляемый прогрессом скролла. Это не набор DOM-элементов и не SVG. Логику движка нужно перенести как есть (см. ниже), обернув в React-хук/компонент.

## Fidelity
**High-fidelity (hifi).** Финальные цвета, типографика, отступы, интерактив. Воссоздавать пиксель-в-пиксель средствами целевой кодовой базы. Единственное, что намеренно «заглушка», — три слота под скриншоты кейсов (см. Assets).

## Технологический контекст прототипа
- Прототип написан как один HTML-файл `ЛАЙМ.dc.html`. Вся логика — в одном JS-классе (рендер города, скролл-трекинг, reveal-анимации, счётчики).
- Шрифты подключены с Google Fonts.
- Никаких внешних JS-библиотек (no GSAP, no Three.js) — город нарисован вручную на 2D canvas. При переносе можно оставить ванильный canvas или взять любой тонкий слой; тяжёлый 3D-движок не нужен.
- `image-slot.js` в прототипе — это drag-&-drop плейсхолдер под картинку для демо. **В проде не нужен** — замените на обычный `<img>` / `next/image`.

## Interactions & Behavior

### A. Живой город (главный эффект) — `<canvas id="lime-city">`
Изометрический 2D-движок на canvas, **управляемый `progress` (0..1) = позиция скролла страницы**.
- **Изопроекция**: `iso(gx, gy, z) = [(gx-gy)*30, (gx+gy)*15 - z]` (tile 30×15, 2:1).
- **Генерация города**: детерминированный PRNG (mulberry32, seed=11). 5 «районов» (foci) с координатами и базовым числом построек `[8,10,12,13,14]`. Типы объектов: `box` (здание, 3 грани), `tower` (градирня — два безье + эллипс-обод), `turbine` (мачта + 3 вращающиеся лопасти), `solar` (наклонные панели сеткой), `gear` (шестерёнка-эллипс с зубьями, вращается). Плюс `connectors` — тонкие Г-образные «энергопути» между постройками с бегущим лаймовым импульсом.
- **Построение по скроллу**: у каждого объекта есть порог `at` (0..1). Высота/прозрачность растут от 0→1 по `easeOutCubic((progress - at)/0.07)`. Район 0 имеет `at≈0` (немного видно сразу), дальше — позже. Итог: пустой старт → плотный «механизм» к концу.
- **Камера**: `camFocus(progress)` интерполирует фокус между 5 foci (easeInOut), `camX = W*0.5 - iso(focus).x`, `camY = H*0.6 - iso(focus).y`. Создаёт «полёт по карте».
- **Постоянная анимация**: лопасти турбин и шестерёнки крутятся по `time` (rAF), импульсы бегут по connectors. Painter's algorithm: сортировка по `gx+gy`.
- **Перенос**: вынести движок в чистый модуль `cityEngine(canvas, getProgress)`; в React — `useRef` на canvas + `useEffect` с rAF. **Прогресс читается каждый кадр** внутри rAF (не только по событию scroll) — это важно для надёжности в разных окружениях.

### B. Скролл-трекинг (важно при переносе)
- `progress = scrollTop / (scrollHeight - innerHeight)`, clamp 0..1.
- Читать позицию из **нескольких источников** (`window.pageYOffset || document.scrollingElement.scrollTop || document.body.scrollTop`) — окружения по-разному назначают скролл-контейнер. В прототипе прогресс пересчитывается **каждый кадр в rAF** + слушатели `scroll` на `window` и `document` (capture). В нормальном Next.js-приложении достаточно обычного `window` scroll, но кадровый поллинг оставьте как страховку.
- **Не делайте `body` скролл-контейнером**: `overflow-x:hidden` на body превращает overflow-y в `auto` и ломает оконный скролл — используйте `overflow-x:clip`.

### C. Reveal-плашки («выезжают со сторон»)
- Элементы с `data-slide="left|right|up"` стартуют `opacity:0` + `translateX(±70px)` / `translateY(46px)`, `transition: opacity 1s / transform 1.05s cubic-bezier(.16,1,.3,1)`.
- IntersectionObserver (threshold 0.16, `rootMargin:'0px 0px -7% 0px'`) при появлении ставит `opacity:1; transform:none` и `transition-delay` из `data-delay` (стаггер).
- Карточки услуг/процесса/кейсов/FAQ выезжают поочерёдно слева/справа/снизу.

### D. Счётчики
- Элементы `[data-count]` (+ опц. `data-prefix`, `data-suffix`) анимируются 0→target за 1500ms, `easeOutCubic`, запуск по IntersectionObserver (threshold 0.6, один раз).

### E. Прочее
- Прогресс-бар сверху: ширина = `progress*100%`.
- Dot-grid: лёгкий параллакс `translateY(-scrollY*0.035)`.
- Hover-состояния на nav, кнопках (подъём + тень), таблетках.
- Аккордеон — нативные `<details>`.

## State Management
Состояния как такового мало — почти всё производное от scroll:
- `progress: number` (0..1) — единственный «живой» стейт, источник правды для города и камеры.
- Reveal/счётчики — одноразовые триггеры через IntersectionObserver (можно хранить `Set` уже показанных).
- Tweak-параметры прототипа (если хотите вынести в настройки/тему): `accent` (цвет), `cityDensity` ('Разреженный'|'Средний'|'Плотный' → множитель числа построек 0.62 / 1 / 1.4), `liveBackground` (bool — выключает canvas).
- Форма: обычный controlled-стейт + ваш submit-handler/валидация (в прототипе submit задизейблен).
